# UmuOS-0.2.x系 基本方針

この文書は、UmuOS 0.2.x 系で何を研究対象とし、何をホスト OS に任せ、何を UmuOS 側で担うかを固定するための方針書である。

0.2.x 系は、`0.1.7-base-stable` で得られた知見を土台にしつつ、RockyLinux 9.7 上の QEMU で UmuOS を観測・運用する系列として位置づける。

---

## 0. 基本姿勢

- 0.2.x 系は「RockyLinux 9.7 上で動く UmuOS」を研究対象とする。
- RockyLinux 9.7 はホストであり、UmuOS は QEMU ゲストとして動く。
- 目的は RockyLinux 9.7 を置き換えることではなく、RockyLinux 9.7 を復旧拠点として残したまま UmuOS を安全に観測・改修できるようにすることである。
- 0.2.x 系では、機能数よりも、詰みにくさ、観測しやすさ、再試行しやすさを優先する。

---

## 1. 0.2.x 系で重視するもの

- 起動経路が理解できること。
- 失敗時に RockyLinux 9.7 から介入できること。
- ネットワーク公開面をホスト側で一元管理できること。
- 生成物、起動方法、観測点が固定されていること。
- 次版へ改善課題を切り出せるように、完成条件を明確に区切ること。

---

## 2. 何をやるか

- Ubuntu で kernel、BusyBox、initramfs、ISO、`disk.img`、起動スクリプトを生成する。
- RockyLinux 9.7 上で QEMU を起動する。
- QEMU ゲスト内部で initramfs から UmuOS rootfs へ `switch_root` する。
- post-switch_root の世界、すなわち `/sbin/init` → `inittab` → `rcS` → サービス起動を研究対象とする。
- QEMU user-net + hostfwd + virtio-net により、グローバル到達可能な観測環境を作る。
- telnet、FTP、アクセスログ、起動観測を、0.2.x 系の実運用に近い最小構成として扱う。

---

## 3. 何をやらないか

- RockyLinux 9.7 の通常運用や通常ブートを壊す変更。
- 実機起動方式への過度な追従。
- UmuOS 独自カーネル常用を 0.2.x の中心テーマにすること。
- Rocky 相当の防御層を UmuOS 側へ持ち込むこと。
- 1 つの版に全課題を詰め込み、完成条件を曖昧にすること。

---

## 4. 責務分離の原則

### 4.1 RockyLinux 9.7 に任せるもの

- QEMU 実行環境
- KVM 利用可否
- SSH、firewalld、ログイン経路などのホスト管理面
- グローバル公開面の入口
- ゲスト障害時の復旧拠点

### 4.2 UmuOS に任せるもの

- initramfs の `/init`
- rootfs 探索と `switch_root`
- `/sbin/init` 以降のユーザーランド
- `rcS` による初期化
- telnet、FTP、ログ、起動観測などの研究対象サービス

### 4.3 文書に任せるもの

- 基本方針書は、0.2.x 系全体の採用原則と版の切り分けを定める。
- 基本設計書は、各版の責務境界と観測対象を定める。
- 詳細設計書は、唯一の正として具体手順と固定値を定める。

---

## 5. 採用する成立パターン

### 5.1 ストレージ方式

- rootfs は ext4 の `disk.img` として管理する。
- QEMU では `disk.img` を virtio disk としてゲストへ渡す。
- ゲスト内部では `/dev/vda` 系のブロックデバイスとして扱う。
- initramfs の `/init` はブロックデバイス探索により rootfs を見つける。

この方式により、開発時はファイルとして管理しやすく、起動時はブロックデバイスとして素直に扱える。

### 5.2 ネットワーク方式

- QEMU user-net
- hostfwd
- virtio-net

この方式を採用する理由は以下の通りである。

- 1 つのグローバル IP でホストとゲストを共存できる。
- RockyLinux 9.7 の SSH を残したまま UmuOS を公開できる。
- ブリッジや TAP の複雑な保守を避けられる。
- 公開面を RockyLinux 9.7 の firewalld と QEMU 引数で一元管理できる。
- トラブル時にホストから直接介入できる。

### 5.3 起動方式

- 0.2.x 系では、まず BIOS 起動で十分とする。
- 大事なのは、QEMU 上で安定して起動し、ゲスト内部で `switch_root` 後のユーザーランドが立ち上がることである。
- UEFI や実機起動互換性は、必要になった段階で別テーマとして扱う。

---

## 6. 成果物方針

Rocky 側へ持ち込む成果物は、原則として次の 3 点に固定する。

- ブート ISO
- 永続 rootfs 用 `disk.img`
- 起動スクリプト

この 3 点方式を維持する理由は以下の通りである。

- Rocky 側の作業手順を最小化できる。
- 失敗時の切り分けがしやすい。
- 再生成、再転送、再起動のループを回しやすい。
- 詳細設計書をコピペ運用しやすい。

---

## 7. セキュリティ運用方針

- telnet と FTP は平文であることを前提に使う。
- 0.2.x 系で重視するのは、絶対安全ではなく、研究対象として扱える現実的な運用可能性である。
- 可能なら Rocky 側または上流で送信元制限を行う。
- 露出面は必要最小限のポートに限定する。
- 復旧経路は必ずホスト側に残す。

---

## 8. ロールバック原則

- RockyLinux 9.7 を常に復旧拠点として残す。
- 実験は QEMU プロセス単位で停止できるようにする。
- 問題が起きても、ホスト OS へ戻って再試行できる設計を保つ。
- ゲスト側で詰んでも、ホスト側からログ確認、プロセス操作、成果物差し替えができる状態を維持する。

---

## 9. 0.2.1-dev の位置づけ

0.2.1-dev は、0.2.x 系の最初の実装固定版であり、次の範囲を完成扱いとして固定した版である。

- RockyLinux 9.7 上での QEMU 起動
- telnet 接続
- FTP ログインと upload/download
- アクセスログ
- 永続 rootfs と起動観測

0.2.1-dev で固定した方式は以下である。

- RockyLinux 9.7 をホストに使う。
- BIOS 起動の QEMU を使う。
- QEMU user-net + hostfwd + virtio-net を使う。
- ISO + `disk.img` + 起動スクリプトの 3 成果物方式を使う。
- rootfs を ext4 の `disk.img` として管理する。

0.2.1-dev で固定しないものは以下である。

- 日本語入力と日本語ファイル名の完全対応
- 露出サービスの最終形
- 将来のセキュア化手段
- UEFI 化や別起動方式への展開
- `0.1.7-base-stable` の補助機能の全面継承

---

## 10. 版の進め方

- 1 つの版には、完成条件を明確に切った範囲だけを載せる。
- その版で未解決の課題は、無理に混ぜず次版課題として切り出す。
- 詳細設計書で実証できたものを、その版の完成条件として認める。
- 次版は、前版の完成条件を壊さず、課題を追加で解く形で進める。

---

## 11. 文書体系

- 基本方針：この文書。0.2.x 系全体の方向性と採用原則を定める。
- 基本設計：版ごとの責務境界、構成、観測対象を定める。
- 詳細設計：実行の唯一の正として、具体手順、固定値、生成方法、確認方法、トラブルシュートを定める。

0.2.x 系では、詳細設計書を唯一の正とし、基本方針書と基本設計書はその判断基準を与える文書として保守する。
