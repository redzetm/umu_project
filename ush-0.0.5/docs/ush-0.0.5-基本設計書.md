# ush-0.0.5-基本設計書.md
UmuOS User Shell (ush) — 基本設計書（0.0.5 / 拡張）  
Target OS: UmuOS-0.1.7-base-stable（想定）

本書は [ush-0.0.5-詳細設計書.md](ush-0.0.5-詳細設計書.md) を唯一の正として矛盾がないように整理した、基本設計（構成・責務・入出力・例外）である。

仕様の要約は [ush-0.0.5-仕様書.md](ush-0.0.5-仕様書.md) に置く。

0.0.5 は 0.0.4 をベースに「普段使いで困る」ための追加仕様（`$(...)`、位置パラメータ、スクリプト引数）を導入する。

---

# 1. 目的と非目的

## 1.1 目的
- UmuOS におけるユーザー向け対話シェルとして、軽量で観測可能な操作環境を提供する。
- 0.0.5 では以下を追加する。
  - コマンド置換 `$(...)`
  - 位置パラメータ（`$0` `$1`..`$9` `$#`）
  - スクリプトモードでの引数受け渡し（`ush script.ush a b`）

## 1.2 非目的
- POSIX 互換の追求。
- ジョブ制御、制御構文、多段パイプ、ヒアドキュメント、高度な FD 操作。
- `$@`/`$*` の複数語展開や `shift` などの「引数処理一式」。
- コマンド置換の入れ子やバッククォート互換。

---

# 2. 実行環境と運用

## 2.1 位置づけ
- `/umu_bin/ush` : ユーザー向け対話操作・軽量スクリプト（本仕様）
- `/bin/sh` : 互換性重視のスクリプト実行（BusyBox ash）

## 2.2 起動・配置
- バイナリの配置先は `/umu_bin/ush` を想定する。
- スクリプト実行（shebang）は `#!/umu_bin/ush` を想定する。

## 2.3 プロンプト設定（環境変数）
- ush はプロンプト表示のたびに `USH_PS1` → `PS1` → デフォルトの順に参照する（0.0.4 継承）。
- ush 自身は rc（例: `/etc/profile`）を自動で読まない。

---

# 3. 全体構成

## 3.1 対話モード処理フロー（概略）
1. 親プロセスの SIGINT 方針を設定（親は落ちない）
2. プロンプトを生成・表示
3. 行編集付きで 1 行入力を取得
4. 空行/コメント行なら何もせず戻る
5. トークナイズ（クォート、演算子、コメント、最小エスケープ、未対応検出）
6. 構文解析（`|` / `&&` / `||` / `;` / リダイレクト制約を検査し、ASTへ）
7. 実行（builtins または外部実行、`last_status` 更新）
8. ループ継続（EOF なら `exit(last_status)`）

補足：
- コマンド置換 `$(...)` は「展開」の一部として実行を伴う。

## 3.2 スクリプトモード処理フロー（概略）
0.0.4 継承に加え、スクリプト引数を保持する。

- `ush <script> [args...]` でファイルを開き、先頭 `#!` 行は無視し、以降 1 行ずつ評価する。
- エラーでも次の行へ進む（fail-fast なし）
- `exit` builtin が実行されたらその時点で終了
- `args...` は位置パラメータとして参照できる

## 3.3 モジュール分割（責務）
0.0.4 の分割を維持し、0.0.5 の追加点を各モジュールに反映する。

- main
  - 対話/スクリプトの分岐、ループ、状態（`last_status` と位置パラメータ）管理、SIGINT 方針
- prompt
  - `USH_PS1`/`PS1`/デフォルト選択、最小展開（`\u` `\w` `\$` `\\`）
- lineedit
  - raw mode 入力、カーソル移動、削除、履歴、Tab 補完
- tokenize
  - 文字列→トークン列（クォート処理、演算子トークン化、コメント除去、最小エスケープ、未対応検出）
  - 0.0.5: `$(...)` を許可するための例外処理（`(` `)` の扱い）
- parse
  - トークン列→AST（`PIPELINE`/`AND`/`OR`/`SEQ`）
- expand
  - 変数展開（`$VAR`/`$?`/`${VAR}`）
  - 0.0.5: 位置パラメータ（`$0` `$1`..`$9` `$#`）
  - 0.0.5: コマンド置換（`$(...)`）
  - チルダ展開（`~`/`~/...`）
- builtins
  - `cd` `pwd` `export` `exit` `help`
- exec
  - AST 評価（短絡、順次実行 `;`、パイプ、リダイレクト、PATH 探索、fork/exec/wait）
  - 0.0.5: コマンド置換のために「標準出力を捕捉して 1 行評価する」補助機能を提供する

# 4. データ設計

## 4.1 状態（`ush_state_t` の拡張方針）
0.0.4 の `last_status` に加えて、位置パラメータ用の引数を保持する。

- `last_status`: 直前の終了コード
- `script_path`（または `$0` 相当）
- `argc/argv`（`$1`.. の元データ）

要件:
- 対話モードでは「引数なし」を表現できる
- スクリプトモードでは `main()` の `argv` を安全に参照できる（寿命は main と同じ）

## 4.2 展開コンテキスト（`ush_expand_ctx_t`）
0.0.4 は `last_status` のみを持つが、0.0.5 では位置パラメータとコマンド置換の実装に必要な情報を追加する。

- `last_status`: `$?` 展開
- `positional`: `$0` `$1`..`$9` `$#` 展開
- `exec hook`: `$(...)` の評価（1行評価 + stdout 捕捉）

# 6. 字句設計（tokenize）

## 6.1 `(` `)` の扱い（0.0.5 追加）
0.0.4 では `(` `)` を未対応として検出していたが、0.0.5 では `$(...)` 導入のため例外を設ける。

- 未クォートで `$(` を検出した場合、その `$(...)` は WORD トークンの一部として許可する
  - ダブルクォート内は、そもそも空白で分割されず 1 WORD に入るため特別扱いは不要（展開で `$(...)` を処理する）
- 未クォートの上記以外の `(` `)` は従来どおり `unsupported syntax`
  - クォート内（`'...'` / `"..."`）の `(` `)` は文字として許可される

設計上の狙い:
- `$(...)` を「演算子」ではなく「WORD の一部」として扱うことで、既存のパーサ（7章）を変えずに導入できる

# 8. 展開設計（expand + glob）

## 8.1 位置パラメータ
- `$0` `$1`..`$9` `$#` を展開対象に追加する
- 未対応（検出してエラー）
  - `$10` 以降
  - `$@` `$*`

## 8.2 コマンド置換 `$(...)`
- `$(cmdline)` の `cmdline` を ush の 1 行として評価する
- stdout を捕捉して文字列化し、WORD へ埋め込む
- 出力の `\r` `\n` はスペースへ正規化してよい（仕様書 11.4.2）
- 入れ子は未対応として検出する

## 8.3 グロブ展開
0.0.4 継承。

# 9. 実行設計（exec）

## 9.1 コマンド置換の stdout 捕捉
`$(...)` を expand が扱う都合上、exec は次を提供する。

- 1 行文字列の評価（tokenize→parse→exec）を「子プロセス」で行う
- その子プロセスの stdout を親が読み取って文字列化する
- 親は `last_status` を上書きしない

補足:
- 捕捉対象は stdout のみ。stderr は親の stderr に流してよい（観測容易性を優先）。

# 11. 受け入れ（基本設計観点）

- `ush script.ush a b` のとき、スクリプトから `$1` が `a`、`$2` が `b` として参照できる
- `echo $(echo hi)` が期待どおり動作する（stdout捕捉 + 正規化）
- `$(...)` 入れ子や `$10` を検出して `unsupported syntax` にできる
